# ADR: 주요 기술적 결정사항

## ADR-001: Android Native Kotlin 사용
결정: Kotlin Android Native로 구현한다.

이유:
- 접근성 서비스, 오버레이, TTS, STT, 앱 실행 intent를 가장 직접적으로 제어해야 한다.
- 크로스플랫폼 프레임워크는 핵심 권한/오버레이/접근성 제어에서 리스크가 크다.

## ADR-002: 서버, 로그인, 서버 DB 없음
결정: V1은 서버와 로그인을 만들지 않는다.

이유:
- 고령 사용자와 가족의 신뢰를 얻는 것이 핵심이다.
- 개인정보/화면 텍스트/음성 원문이 외부로 나가지 않는다는 약속을 단순하게 지킨다.
- 출시 속도를 높인다.

## ADR-003: AI API 미사용
결정: V1 핵심 경로에 AI API를 쓰지 않는다.

이유:
- 1초 이내 안내 목표와 네트워크 지연은 충돌한다.
- 화면/음성 데이터가 외부로 나가는 순간 신뢰 메시지가 약해진다.
- 좁은 범위에서 규칙 기반 + 노드 탐색이 더 안정적이다.

향후:
- 사용자가 명확히 동의한 "더 똑똑한 큐" 모드에서만 검토 가능.

## ADR-004: DB 대신 DataStore + assets JSON
결정: Room DB 없이 `Preferences DataStore`와 `assets JSON`만 사용한다.

이유:
- 저장할 데이터가 설정뿐이다.
- 앱별 사전은 정적 번들로 충분하다.
- MVP에서 DB 마이그레이션과 스키마 복잡도를 피한다.

## ADR-005: 좌표 레시피 금지
결정: 앱별 고정 좌표를 저장하지 않는다.

이유:
- 기기 해상도, 글자 크기, 앱 업데이트에 취약하다.
- 확장성이 낮다.

대안:
- 접근성 노드의 텍스트, contentDescription, className, actions, bounds를 사용한다.
- 앱별 데이터는 좌표가 아니라 행동/키워드/금지어/상태 힌트만 둔다.

## ADR-006: 상태 기반 탐색 + 점수 기반 후보 선택
결정: 단순 점수 기반이 아니라 `ScreenContext`를 먼저 판단한 뒤 후보를 찾는다.

이유:
- 전체 노드에서 직접 찾으면 후보가 많아져 성공률 90%를 보장하기 어렵다.
- 화면 상태를 좁히면 잘못된 후보를 줄일 수 있다.

## ADR-007: 자동 클릭 금지
결정: 큐는 자동 클릭하지 않는다. 사용자가 직접 누른다.

이유:
- 접근성 정책 리스크를 줄인다.
- 고령 사용자와 가족의 신뢰를 지킨다.
- 결제/송금/전송 사고를 예방한다.

## ADR-008: 흰색 마스킹은 조각 오버레이로 구현
결정: 사용자에게는 전체 흰색 마스크처럼 보이게 하되, 구현은 후보 영역을 비운 흰색 사각형 오버레이 조합으로 한다.

이유:
- 전체 오버레이에 구멍을 뚫는 방식은 터치 전달이 기기별로 예민할 수 있다.
- 후보 영역에 오버레이를 두지 않으면 아래 앱이 직접 터치를 받는다.

## ADR-009: 마스킹 중 화면 텍스트 없음
결정: 마스킹 상태에는 텍스트 안내를 표시하지 않는다.

이유:
- 텍스트가 버튼이나 중요한 화면을 가릴 수 있다.
- 고령 사용자가 시각적으로 볼 것은 "남겨진 곳" 하나면 충분해야 한다.

보완:
- TTS로만 "눌러주세요." 안내.
- 좌측 상단 화살표로 종료 가능.

## ADR-010: 자동 입력은 제한적으로 허용
결정:
- 쿠팡 검색어 자동 입력 허용.
- 카톡 메시지 본문 자동 입력 허용.
- 민감 정보 자동 입력 금지.

이유:
- 타이핑은 고령 사용자에게 큰 장벽이다.
- 검색어와 메시지 입력은 체감 가치가 크다.
- 단, 보내기/결제/송금/인증 관련 행동은 안내하지 않는다.

## ADR-011: 릴리즈 로그 저장 안 함
결정: 릴리즈 빌드에서 화면 텍스트, 음성 원문, 명령 로그를 저장하지 않는다.

이유:
- 신뢰 메시지를 단순하게 유지한다.
- 개인정보 리스크를 낮춘다.

테스트:
- 성공률 측정은 수동 테스트와 개발 빌드 도구로 수행한다.

## ADR-012: 앱 업데이트 대응은 앱 업데이트로 한다
결정: 외부 앱 UI 변경에 따른 사전 수정은 우리 앱 업데이트로 배포한다.

이유:
- 서버 없는 원칙과 맞다.
- MVP 단계에서 원격 사전 업데이트 인프라는 과하다.

확장 전략:
- 의미 기반 `TargetType`과 얇은 앱별 사전으로 새 앱 추가 비용을 낮춘다.

## ADR-013: 성공률 기준은 지원 범위 안에서 정의
결정: 90% 성공률은 "모든 앱 모든 요청"이 아니라 "지원 앱 3개, 지원 명령 6개 이하" 범위 안에서 측정한다.

이유:
- 좁고 확실한 성공이 YC MVP에 더 적합하다.
- 실패할 수 있는 범위를 솔직히 제한해야 제품 신뢰가 유지된다.
